# F-7 — Mutable-borrow lifetime / mode operational semantics (CRA-200)

## What changed
Added `crawfish/borrow.py` — the borrow semantics + concurrency module. It implements
a **dynamic exclusive borrow with a Store-backed atomic acquire**, downgrading ALG-4's
unenforceable "statically unaliasable" claim (see ADR 0018). No existing files were
edited; the module is self-contained and touches only the **public** `Store` API.

## Public API added
```python
# crawfish/borrow.py
class ExclusiveBorrowError(RuntimeError): ...      # raised on a concurrent acquire

@runtime_checkable
class Borrowable(Protocol):                        # anything with a stable .id
    @property
    def id(self) -> str: ...

class Borrow:                                      # the live handle yielded by mutable()
    target: Borrowable
    epoch: int                                     # acquisition epoch (diagnostics)
    active: bool
    def release(self) -> None: ...                 # idempotent

def borrow_key(target: Borrowable, epoch: int) -> str          # "borrow:<id>:acq:<epoch>"

@contextmanager
def mutable(target: Borrowable, store: Store, *, org_id: str = "local") -> Iterator[Borrow]:
    ...                                            # acquires on enter, releases on exit
```
Also exports `BORROW_RECORD_KIND = "borrow_lock"` (the record namespace this module owns).

## How it works (claim-key scheme)
- Borrow key: **`borrow:<defn_id>:acq:<epoch>`**, deterministic and identity-derived.
  Tenancy is applied by the Store via `org_id`, so the same id in two orgs → two claims.
- `claim_idempotency` is the atomic gate (`store/sqlite.py:107`, reused per `nodes/sink.py:143`).
- An **epoch** in a `borrow_lock` record makes the single-shot claim re-acquirable:
  acquire claims `acq:<e>` and writes `{held:true, epoch:e}`; **release** writes
  `{held:false, epoch:e+1}` so the next acquire claims a fresh `acq:<e+1>`. No claim is
  ever re-used or deleted — release needs only `put_record`, so **no Store edit was required**.
- Release is idempotent; the context manager's `finally` releases even on exception.

## Request to other owners — wire `Definition.mutable()`
`definition/types.py` is owned by another agent this wave, so the borrow is exposed as a
free function `mutable(defn, store, *, org_id=...)`. **Follow-up:** add a thin delegating
method to `Definition`:
```python
# in crawfish/definition/types.py, on class Definition
def mutable(self, store: Store, *, org_id: str = "local") -> AbstractContextManager[Borrow]:
    from crawfish.borrow import mutable as _mutable
    return _mutable(self, store, org_id=org_id)
```
`Definition.id` already satisfies the `Borrowable` protocol — no other change needed.
Tests in `test_borrow.py` call the `borrow.py` API directly and do not depend on this wiring.

## SECURITY note
The borrow is **tenancy-scoped by construction**: both the idempotency key and the
`borrow_lock` record carry `org_id`, so a borrow held by org `a` never blocks (or is
visible to) org `b` — proven by `test_cross_tenant_does_not_block` against a real
`SqliteStore`. Enforcement lives in the Store, so the exclusivity guarantee holds across
processes (not just async tasks) and survives the SQLite→Postgres driver swap unchanged.

## Tests
`packages/crawfish/tests/test_borrow.py` — **10 tests**, all against a real on-disk
`SqliteStore` (`tmp_path`), no mocks. Covers: sequential concurrent-acquire raises;
async two-task race (exactly one wins, one `ExclusiveBorrowError`); acquire/release/
re-acquire round-trip with epoch advance; cross-tenant non-blocking; distinct-definition
non-blocking; release-on-exception; idempotent release; deterministic key; record state.
