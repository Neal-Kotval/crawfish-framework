# F-2 — Loop/program ledger schema, composite key (CRA-195)

Owner: `packages/crawfish/src/crawfish/ledger.py` (+ one additive migration in
`store/migrations.py`). **NEW key space, not a reuse of the linear pipeline ledger.**

## What changed

The linear pipeline ledger — `checkpoint_step(pipeline_id, step_index: int)` /
`completed_steps -> set[int]` — cannot represent per-`(item, edge, visit)` progress or a
per-item recursion depth stack. F-2 adds an **explicit extended ledger key space** that
lives alongside it (the pipeline ledger is untouched and keeps working).

### New composite key space (a SEAM change — flag for ARCHITECTURE.md)

Two coordinate families, both recording the iteration's frozen Output reference:

- **loop / back-edge visits:** `(loop_id, item_id, edge_id, visit) -> output_ref`
- **recurse depth stack:** `(loop_id, item_id, depth) -> output_ref`

Persisted under a **new record kind `ledger_loop`** in the generic `records` table (no
new table — the namespace is a new *kind*). The existing `ledger_pipeline`,
`ledger_item`, and `ledger_run` kinds are unchanged. All raw persistence stays inside the
Store: `ledger.py` only orchestrates the generic `put_record` / `get_record` /
`list_records` API. No raw SQL was added to `ledger.py`.

New `ExecutionLedger` methods (mirroring the existing API style):

```python
checkpoint_iteration(loop_id, item_id, edge_id, visit, output_ref) -> None
completed_visits(loop_id, item_id, edge_id) -> set[int]
iteration_output_ref(loop_id, item_id, edge_id, visit) -> str | None
# recurse variant:
checkpoint_depth(loop_id, item_id, depth, output_ref) -> None
completed_depths(loop_id, item_id) -> set[int]
depth_output_ref(loop_id, item_id, depth) -> str | None
```

### `compute_loop_id` — deterministic, never `new_id()`

```python
loop_id = sha256( v || body_version_sha || item_lineage || edge_id )   # length-prefixed
```

A pure module-level function `compute_loop_id(body_version_sha, item_lineage, edge_id)
-> str`. **Hard requirement:** a loop's id is *derived*, never minted with `new_id()`, so
two independent process invocations of the same loop body over the same item along the
same back-edge re-derive the **same** id — and resume re-charges **$0** for iterations
already recorded. Inputs are length-prefixed (and version-tagged, `_LOOP_ID_VERSION = 1`)
so distinct concatenations cannot collide.

### output_ref uses F-0's `output_content_sha`

Each iteration pins the **content sha** of its frozen `Output` (from F-0,
`crawfish.output.output_content_sha`) — a stable digest over an immutable value, not a
volatile per-instance `id`. Resume can compare/restore by content identity.

### org_id on every row (flag for SECURITY.md)

Every `ledger_loop` row carries `org_id` (reused from `ExecutionLedger._org`). Reads
scope to the instance's org, so a resume in org "b" sees **none** of org "a"'s completed
iterations. Cross-tenant resume cannot replay another org's work. (Tested:
`test_cross_org_isolation`, `test_depth_cross_org_isolation`.)

## Migration

`store/migrations.py` adds **Migration 3** (`CURRENT_SCHEMA_VERSION` -> 3):
`CREATE INDEX IF NOT EXISTS idx_records_org_kind ON records(org_id, kind)`. The
`completed_visits` / `completed_depths` scans resolve completed iterations by
`list_records("ledger_loop", org_id=...)`, so the leading `(org_id, kind)` index keeps
that scan sargable as the loop ledger grows. Additive and idempotent per the ADR 0014
authoring contract; existing migrations were not altered. **Store files were NOT touched**
— the generic record API (`put_record`/`get_record`/`list_records`) was sufficient; only
`migrations.py` gained the new entry (its owned file).

## ADR note

The content-hash / schema-versioning ADR (ADR 0014, and the F-0/F-1 hash-schema thread)
should cover this composite key space: the `ledger_loop` key encoding, `_LOOP_ID_VERSION`
(bump = migration of every derived `loop_id`), the use of F-0's `output_content_sha` as
the output_ref, and the forbidding of `new_id()` for loop identity.
