# F-4 — `GoldenSet.from_corrections` + a `correction` ledger event kind (CRA-197)

## What changed
Corrections are now a **first-class, queryable signal**. A new `correction`
emission kind makes the source set explicit on the ledger, and
`GoldenSet.from_corrections` mines that corpus into a `GoldenSet` — the foundation
four dependents (CL-2, TS-3, TS-7, R4) build on. Until they adopt it, every
dependent still accepts an explicitly-authored `GoldenSet` (the uniform fallback,
unchanged).

This reuses the existing generic Store record/event API (the F-2 pattern) — **no
`store/base.py` or `sqlite.py` edits**. The F-3 gate algebra
(`paired_gate`/`precision_gate`/`save_baseline`/…) is untouched.

## The `correction` emission kind
`crawfish.emission` gains:

```python
class EmissionKind(str, Enum):
    ...
    CORRECTION = "correction"   # ground-truth correction (feeds GoldenSets/guards)

REQUIRED_ATTRS[EmissionKind.CORRECTION] = ("correction_type", "provenance")

class CorrectionType(str, Enum):     # the sub-category — HOW the wrong output was found
    HUMAN_REVERT  = "human_revert"
    CI_FAILURE    = "ci_failure"
    REVIEW_REJECT = "review_reject"

class Provenance(str, Enum):         # WHO authored it — the trust class (Security S4)
    TRUSTED   = "trusted"
    UNTRUSTED = "untrusted"
```

The three correction **types** are sub-categories carried in `attrs["correction_type"]`;
they are filterable. Enums are `(str, Enum)` per ADR 0004.

## API added
```python
# crawfish.emission
def emit_correction(
    store, *, run_id, correction_type: CorrectionType, provenance: Provenance,
    org_id="local", tainted=False,
    inputs=None, expected=None, produced=None,
    node_id=None, pipeline=None, ts=0.0, attrs=None,
) -> Emission: ...

def read_corrections(
    store, *, org_id="local", kinds: tuple[CorrectionType, ...] | None = None,
) -> list[Emission]: ...

CORRECTION_RECORD_KIND = "correction"   # Store record namespace for the corpus

# crawfish.eval — classmethod on GoldenSet
@classmethod
def from_corrections(
    cls, store, name="corrections", *, org_id="local", version="0.1",
    kinds=(CorrectionType.HUMAN_REVERT, CorrectionType.CI_FAILURE, CorrectionType.REVIEW_REJECT),
) -> GoldenSet: ...
```

`emit_correction` writes the correction **twice**: as a typed `Emission` on the
run's ledger (the taxonomy half — per-run, taint-propagating) **and** as a generic
Store record under `CORRECTION_RECORD_KIND` keyed by the emission id (the corpus
half — listable across runs, kind-filterable, org-isolated). `read_corrections` is
the cross-run query (unlike `read_emissions`, which is per-run); it filters by
`CorrectionType`.

## SECURITY — provenance / taint gate (flag SECURITY.md: Gap S4 corpus poisoning)
Corrections feed guards/verifiers as **ground truth**, so a poisoned correction
corpus is an attack surface (Security Gap S4). The control is explicit and lives in
`from_corrections`:

- **WHO may emit** is declared on every correction via `provenance`
  (`TRUSTED` / `UNTRUSTED`). A correction authored from fluid/untrusted session data
  must be marked `UNTRUSTED`.
- **Taint propagation** uses the existing `Emission.tainted` marker: a correction
  carrying any value derived from FLUID input stays tainted across the emission
  boundary (the corrected→guard path is taint-analyzed).
- **The admission gate is an AND**: a correction enters the GoldenSet as trusted
  ground truth **only if** `provenance == TRUSTED` **AND** `tainted is False`.
  Anything `UNTRUSTED` **or** fluid-tainted is **quarantined** — it stays on the
  ledger for audit but never becomes a guard's ground truth. The AND is load-bearing:
  a fluid-derived value cannot silently become ground truth even if mislabelled
  `TRUSTED` (taint wins).

`emit_correction` itself records **every** attempt (audit completeness); the trust
decision is made at admission, not at write, so quarantined corrections remain
inspectable.

## Tenancy / determinism
Every correction record and ledger event carries `org_id` (defaulted `"local"`).
Cross-org isolation holds: `from_corrections(org_id="a")` and `read_corrections`
only see org `a`'s corrections; org `b` is invisible. Deterministic given a fixed
ledger — no clock, no model call; case ids are the emission ids, so the same ledger
always yields the same set.

## The authored-set fallback (unchanged)
`GoldenSet(store, name)` + `.add(EvalCase(...))` still works for dependents that
have not yet adopted `from_corrections`. Pinned by
`test_authored_goldenset_fallback`.

## Tests
`packages/crawfish/tests/test_corrections.py` — 7 tests against a real `SqliteStore`
in `tmp_path`, covering the five acceptance criteria: (1) count matches the ledger;
(2) cross-org isolation; (3) the `correction` kind is queryable/filterable;
(4) provenance/taint gate quarantines untrusted *and* fluid-tainted corrections;
(5) the authored-set fallback. Plus a determinism test (same ledger ⇒ same set).

Two existing taxonomy-freeze tests were extended for the new kind (legitimate
contract growth, not a behaviour change): `test_emission.py::test_every_kind_round_trips`
(adds `correction_type`/`provenance` sample attrs) and
`test_interface_freeze.py::test_emission_kind_taxonomy_is_closed_and_covered`
(adds `"correction"` to the closed set). Full suite: 773 passed, 1 skipped.
