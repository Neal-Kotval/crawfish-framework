# CRA-212 — AL-T5: variance-aware promotion gate

The promotion gate now promotes a candidate only when its gain over the stored
baseline **clears the per-metric noise band**, not merely when it is `> 0`. This
makes the F-3 eval gate noise-robust without changing its hard-rejection
invariant.

## What changed (`crawfish.eval`)

- **`save_baseline(store, name, scores, *, std=None, org_id="local")`** — gained an
  optional `std`. The `scores` record format is **unchanged**: when `std` is omitted
  the call writes exactly the record it always did, so every pre-CRA-212 baseline and
  every existing caller is byte-for-byte back-compatible. When given, `std` is written
  to a **parallel** `eval_baseline_std` record under the same `name`/`org_id` (tenancy
  preserved). `std=None` never erases an existing band.
- **`load_baseline_std(store, name, *, org_id="local") -> dict[str, float] | None`** —
  new. `None` (no std record) is the back-compat signal to treat the band as
  zero-width.
- **`save_baseline_from_report(store, name, report, *, org_id="local")`** — new
  convenience: persists a `CalibrationReport`'s `rubric_mean` as the baseline scores and
  its `rubric_std` as the noise band (respects the report's `org_id`).
- **`promote_against_baseline(...) -> PromotionVerdict`** — new. The variance-aware
  gate. Promotes iff BOTH:
  - **hard gate (F-3, unchanged):** no metric regresses past its noise band
    (`metrics.is_regression_variance_aware` with the recorded `std`) — a candidate that
    maxes one metric while truly regressing another is still rejected; and
  - **improvement clears the band:** the primary metric's gain exceeds `k·std`
    (`metrics.noise_band`, `k` derived from `alpha`).
  Winner's-curse correction (F-8): when a `fresh_sample` is supplied, the promoted
  scores are shrunk toward it (`experiment.winners_curse_shrink`) before being stored as
  the new baseline, so the bar cannot ratchet up on selection noise. The carried-forward
  `std` keeps the new baseline variance-aware. A rejected candidate writes nothing.
- **`PromotionVerdict`** — new frozen dataclass (`promoted` / `regressed` /
  `cleared_band` / `primary` / `primary_gain` / `primary_band` / `reason`), auditable.

## Back-compat

With no std record (or `std == 0`) the band is zero-width: the hard gate reduces
byte-for-byte to `is_regression` and the improvement test to "primary gain > 0" — i.e.
today's single-point behaviour. With no baseline at all, promotion is allowed
(seed-and-promote), mirroring `gate_against_baseline`. The baseline **record format is
unchanged**; std lives in a separate record, so old baselines load identically.

## Determinism / security

Pure arithmetic over recorded scores + recorded std: same inputs ⇒ same verdict. The
promotion decision is derived from trusted measurements (scores/std), never from
fluid/untrusted session data. Both baseline records carry `org_id` (tenancy isolation).

## Decision log — noise-band threshold (k·std vs significance test)

`promote_against_baseline` uses the **k·std band** (`noise_band`, `k` from `alpha`) on
the *recorded* per-metric std, matching this issue's worked example
(0.80±0.05 / 0.82 in-band / 0.92 beyond) and the `metrics.is_regression_variance_aware`
sibling. This is the right gate for the Tuner/learning caller, which retains aggregate
scores + a recorded std but **not** per-case deltas. The paired-bootstrap significance
test (`eval.paired_gate`, gate **b**) remains available for callers that *do* retain
per-case score vectors over the same GoldenSet; the two are the documented arithmetic
vs. paired siblings of the same F-3 algebra.
