# F-0 тАФ `Output` content hashing (CRA-193)

## What changed
Added a pure helper `output_content_sha` to `crawfish.output`. No change to the
`Output` model: it stays frozen (`output.py`, `model_config = {"frozen": True}`),
`derive()` is untouched (still mints a fresh immutable Output with a new `id` and
propagated taint/lineage), and **no `sha` field** was added to the model. This is
the single content-hash primitive that CL-1/CL-4/C2/C3/TS-1/R1/R3 build on for
"mint a new frozen Output sha / no-progress-by-sha / output_sha in ledger".

## Public API added
```python
def output_content_sha(o: Output[object]) -> str: ...
```
Returns a lowercase hex SHA-256 digest (64 chars). Exported from
`crawfish.output` and via `crawfish.output.__all__`. (`__init__.py` was **not**
edited, per the one-owner rule; import from `crawfish.output`.)

## What enters / excludes the hash, and why
Hashed (the fields that define **structural equality**):
`output_schema`, `value`, `produced_by`, `lineage`, `tainted`.

Excluded: `id` тАФ it is a fresh per-instance UUID (`default_factory=new_id`), so
including it would make every Output hash unique and break acceptance #2/#4.
Excluding it makes two structurally-equal Outputs hash equal regardless of `id`.

The digest is over **canonical JSON**: `model_dump(mode="json")` then
`json.dumps(..., sort_keys=True, separators=(",", ":"), ensure_ascii=True)`,
wrapped as `{"v": <version>, "content": {...}}`. Canonicalization makes it
deterministic across processes and order-independent for dict-valued payloads.

## Migration / hash-bump notes
A `_CONTENT_SHA_VERSION` (currently `1`) is folded into the hashed payload. Any
future change to the hashed field set or the canonicalization MUST bump this
constant тАФ it changes every digest, so any persisted ledger keyed on
`output_content_sha` would need a re-key/migration.
